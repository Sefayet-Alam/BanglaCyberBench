Overall verdict

The manuscript has a strong Q1-level foundation: the benchmark construction, multi-source evaluation, script-aware design, cross-source robustness analysis, human annotation study, and LLM comparison form a coherent research story. The class/source/split totals are internally consistent at 94,323; the confusion matrix exactly reproduces the reported Accuracy, Macro-F1, Weighted-F1, MCC, supports, and per-class scores; and the ensemble weights sum to 1.0.

However, I would not submit this version as the final manuscript yet. Several methodological contradictions are likely to be caught immediately by a careful reviewer.

Submission-blocking issues
1. The official test set appears to have been used for model selection

Section 3.5 says the 20% test set was never touched during tuning. However:

Table 15 performs the component ablation on the 20% test split.
The text then uses those test results to choose CE+FGM as the proposed recipe.
Table 17 tests the script-mask decision on the Romanized portion of the test set.
The taxonomy decision is also defended using test-set results.

This contradicts the claim that the test set is a clean, untouched final evaluation.

Required correction: perform model/component/mask/taxonomy selection using validation results only. Freeze every decision before producing the final test results. If the test set has already influenced those decisions, create a genuinely untouched final holdout or use nested cross-validation.

2. BanglaBERT’s training/evaluation protocol contradicts Table 11

Section 4.4 states that BanglaBERT:

“trains, validates, and is evaluated only on Bangla-script rows.”

But Table 11 says all twelve individual models were evaluated on the identical complete 20% test split, and BanglaBERT obtains Macro-F1 up to 0.8154. A Bangla-only model emitting neutral logits for 39.6% Romanized rows cannot simultaneously be a directly comparable full-test single model.

Choose and document one consistent design:

Train BanglaBERT on all rows but mask it only during ensemble fusion; or
Train it only on Bangla rows and report its standalone performance only on the Bangla subset; or
Present a separate full-scope BanglaBERT baseline and a different Bangla-specialist ensemble member.

At present, Table 11 cannot safely support the claim that the ensemble beats every actual member.

3. The cross-source explanation contains a factual error

The manuscript says that holding out BD-SHS leaves the smallest training pool. This is incorrect when BANTh is excluded:

Hold out Facebook-44K: training pool = 8,882 + 5,029 = 13,911
Hold out Multilabel-12.5K: training pool = 43,078 + 5,029 = 48,107
Hold out BD-SHS: training pool = 43,078 + 8,882 = 51,960

BD-SHS actually leaves the largest, not the smallest, source-held-out training pool.

Section 3.5 also says the “remaining sources” form training data, while Table 18 says only the other two Bangla-script sources are used. Explicitly state that BANTh is excluded from both training and testing in this experiment to keep script fixed.

4. The base-paper comparison is not truly like-for-like

The manuscript repeatedly calls this a same-protocol or like-for-like comparison, but:

The base paper uses 44,001 comments.
Your deduplicated source contains 43,078.
It is unclear whether the exact same train/validation/test instances were reproduced.
The reported base-paper scores appear to be taken from the publication rather than generated on your test indices.

Therefore, the claimed Threat improvement of +0.0713 may partly reflect different test samples.

The strongest solution is to rerun the base model on your exact deduplicated split. Otherwise, rename this a protocol-aligned, unpaired literature comparison and avoid wording such as “on the identical split,” “like-for-like,” or “clearly improves Threat.”

5. The ablation does not strictly select CE+FGM as the best recipe

Table 15 reports:

CE+FGM: 0.8071
CE+FGM+R-Drop: 0.8083
CE+FGM+EMA: 0.8093

Thus, CE+FGM is the simplest configuration, but not the highest-scoring configuration. Your complexity argument is reasonable, but it is not demonstrated that the twelve-model ensemble makes EMA or R-Drop redundant.

Either:

Evaluate the ensemble with CE+FGM+EMA/R-Drop; or
State explicitly that CE+FGM was selected as a parsimony-performance trade-off, not as the best ablation result.

Also make every configuration label explicit—for example, “CE+FGM+EMA”—because the current “+EMA” notation is ambiguous.

LLM evaluation: valuable addition, but under-documented

The exact model identifier, experiment date, API/SDK version, full prompt, output schema, thinking configuration, safety settings, retry policy, parsing failures, refusals, and invalid-output count should be reported. The official model code is gemini-2.5-flash; the current reference [36] is a generic 2024 Gemini report and does not properly identify Gemini 2.5 Flash.

The explanation that Gemini overpredicts safe or majority classes is not currently supported by a per-class table. Add its per-class precision, recall, F1, prediction distribution, and preferably a confusion matrix.

Also disclose that the test comments were transmitted to a third-party API and state which paid/logging configuration was used. Google’s current documentation describes separate logging and abuse-monitoring retention processes, including retention of prompts and outputs for safety monitoring; the applicable terms at the date of your experiment should be checked and documented.

Replace:

“We read the zero-shot score as a floor, not a competitor.”

with:

“We treat this result as a single-prompt zero-shot reference rather than a comprehensive evaluation of LLM-based classification.”

Annotation study corrections

The 500-sample study is a meaningful improvement, but add:

Exact stratification: samples per class and script
Sampling seed and procedure
Annotation instructions and examples
Whether disagreements were adjudicated
A confidence interval for Cohen’s κ
Per-class agreement or the annotator confusion matrix
The number of comments on which both annotators disagreed with the consolidated label

Do not call the consolidated labels “real values.” Use benchmark consolidated labels or mapped dataset labels.

There is also a direct textual inconsistency: Section 3.4 says annotator agreement was weakest for Threat and Sexual and that these are the same classes the model finds hardest. Section 6.3 identifies Threat and Abusive as the two hardest model classes.

Because Naim participated in annotation, add “data validation and annotation” to his author-contribution statement.

The ethics declaration says no new data were collected from human participants. Although author annotation of public comments is normally low-risk and may be exempt, the statement should acknowledge the human annotation exercise and state the applicable institutional determination or exemption.

Numerical and reporting corrections
The displayed confidence-interval widths in Table 10 do not match the displayed endpoints:
Metric	Shown interval	Width from endpoints	Reported width
Macro-F1	[0.8151, 0.8298]	0.0147	0.0148

Accuracy	[0.8285, 0.8392]	0.0107	0.0104
MCC	[0.7369, 0.7530]	0.0161	0.0158

Either calculate widths from the rounded endpoints or display more decimal places.

Contribution C6 says all headline metrics carry bootstrap CIs, but Macro-AUROC has no CI in Table 10.
The displayed R-Drop delta is 0.8083 − 0.7961 = 0.0122, not 0.0123, unless unrounded values are used. Show more precision or align the displayed delta.
The five-class taxonomy ablation reports 0.8073, while the CE+FGM component row reports 0.8071. If these represent different runs, seeds, or scopes, state this clearly.
The title of Section 6.5 says “Romanized-Script Accuracy,” although the reported metric is Macro-F1.
The claim that the Romanized score is low chiefly because of script difficulty or data volume is too strong. BANTh is simultaneously the only Romanized source, so source, label-policy, and script effects remain confounded—exactly as the limitations section acknowledges.
Bootstrap intervals quantify uncertainty from the fixed test sample, not uncertainty from retraining. Avoid using them alone to claim training stability.
Optimisation and reproducibility inconsistencies

Equation 14 includes decoupled weight decay λ, but Algorithm 1 says “no decay.” Table 8 says “uniform, no decay” beside two different encoder/head learning rates.

Report one consistent configuration including:

Weight decay value

Warm-up proportion or number of steps
Scheduler
Gradient-clipping norm
Adam epsilon
Early-stopping metric
Model-selection rule
Maximum number of Nelder–Mead iterations and number of restarts

The classical TF-IDF baseline also needs a setup subsection describing its word/character n-grams, classifier hyperparameters, tuning space, preprocessing, and validation procedure.

Add uniform-logit averaging to the ensemble comparison:

Best validation-selected single model
Uniform masked ensemble
Optimised weighted masked ensemble
Unmasked ensemble

A paired bootstrap CI for the weighted-versus-uniform and ensemble-versus-single differences would substantially strengthen the ensemble claim.

BanglaCyberBench: A Deduplicated Dual-Script Benchmark for Cross-Source Bengali Cyberbullying Detection

TALLIP explicitly covers information processing for Asian and low-resource languages. Bengali/Banglish cyberbullying, dual scripts, heterogeneous corpora, and low-resource evaluation are directly aligned with that mission.
